Exports Qdrant metrics in Prometheus text-based exposition format for monitoring and alerting.

## Description

The `/metrics` endpoint provides comprehensive system and application metrics compatible with Prometheus and other monitoring systems. Metrics are automatically collected from telemetry data and formatted according to the [Prometheus exposition format](https://prometheus.io/docs/instrumenting/exposition_formats/).

## Query Parameters

<ParamField query="anonymize" type="boolean" default="false">
  Anonymize collection names and sensitive data in metric labels
</ParamField>

<ParamField query="timeout" type="integer" default="60">
  Timeout in seconds for collecting metrics (minimum: 1)
</ParamField>

## Response Format

Returns `text/plain` content with Prometheus metrics:

```
# HELP app_info information about qdrant server
# TYPE app_info gauge
app_info{name="qdrant",version="1.8.0"} 1

# HELP cluster_enabled is cluster support enabled
# TYPE cluster_enabled gauge
cluster_enabled 1

# HELP cluster_commit index of last committed operation
# TYPE cluster_commit counter
cluster_commit{peer_id="123456789"} 1234

# HELP collections_total number of collections
# TYPE collections_total gauge
collections_total 5

# HELP collection_points approximate amount of points per collection
# TYPE collection_points gauge
collection_points{id="my_collection"} 1000000
```

## Metric Categories

### Application Metrics

<Accordion title="App Info & Features">
  **app_info** (gauge)
  - Information about Qdrant server version
  - Labels: `name`, `version`
  
  **app_status_recovery_mode** (gauge)
  - Whether recovery mode is enabled (1) or disabled (0)
</Accordion>

### Cluster Metrics

<Accordion title="Cluster Status">
  **cluster_enabled** (gauge)
  - Whether clustering is enabled (1) or disabled (0)
  
  **cluster_peers_total** (gauge)
  - Total number of peers in the cluster
  
  **cluster_term** (counter)
  - Current Raft consensus term
  
  **cluster_commit** (counter)
  - Index of last committed operation
  - Labels: `peer_id`
  
  **cluster_pending_operations_total** (gauge)
  - Number of operations waiting to be applied
  
  **cluster_voter** (gauge)
  - Whether this peer is a voter (1) or learner (0)
  
  **cluster_working_state** (gauge)
  - Consensus thread status
  - Labels: `state` ("working" or "stopped")
  
  **cluster_last_update_timestamp_seconds** (counter)
  - Unix timestamp of last consensus update
</Accordion>

### Collection Metrics

<Accordion title="Collections & Points">
  **collections_total** (gauge)
  - Total number of collections
  
  **collections_vector_total** (gauge)
  - Total number of vectors across all collections
  
  **collection_points** (gauge)
  - Approximate point count per collection
  - Labels: `id` (collection name)
  
  **collection_vectors** (gauge)
  - Vector count by vector name
  - Labels: `collection`, `vector`
  
  **collection_indexed_only_excluded_points** (gauge)
  - Points excluded from indexed-only requests
  - Labels: `id`, `vector`
</Accordion>

<Accordion title="Collection Health">
  **collection_active_replicas_min** (gauge)
  - Minimum active replicas across all shards
  
  **collection_active_replicas_max** (gauge)
  - Maximum active replicas across all shards
  
  **collection_dead_replicas** (gauge)
  - Total shard replicas in non-active state
  
  **collection_running_optimizations** (gauge)
  - Number of running optimization tasks
  - Labels: `id`
  
  **collection_update_queue_length** (gauge)
  - Pending operations in update queue
  - Labels: `id`
</Accordion>

<Accordion title="Shard Transfers">
  **collection_shard_transfer_incoming** (gauge)
  - Number of incoming shard transfers
  - Labels: `id`
  
  **collection_shard_transfer_outgoing** (gauge)
  - Number of outgoing shard transfers
  - Labels: `id`
</Accordion>

## Custom Metric Prefix

You can configure a custom prefix for all metrics in your `config.yaml`:

```yaml
service:
  metrics_prefix: "qdrant_"
```

This results in metrics like:
```
qdrant_collections_total 5
qdrant_cluster_commit{peer_id="123"} 1234
```

## Separate Metrics Port

For security, you can expose metrics on a dedicated port:

```yaml
service:
  host: 0.0.0.0
  http_port: 6333
  metrics_port: 9091  # Separate metrics port
```

Then configure Prometheus to scrape `http://qdrant:9091/metrics`.

## Example Usage

<CodeGroup>
```bash cURL
curl http://localhost:6333/metrics
```

```bash Filter Specific Metric
curl http://localhost:6333/metrics | grep collection_points
```

```bash Anonymized Metrics
curl "http://localhost:6333/metrics?anonymize=true"
```

```yaml Prometheus Scrape Config
scrape_configs:
  - job_name: 'qdrant'
    scrape_interval: 15s
    static_configs:
      - targets: ['qdrant:6333']
```

```python Python
import requests
from prometheus_client.parser import text_string_to_metric_families

response = requests.get("http://localhost:6333/metrics")
for family in text_string_to_metric_families(response.text):
    for sample in family.samples:
        print(f"{sample.name}{sample.labels}: {sample.value}")
```
</CodeGroup>

## Implementation Details

Metrics are generated from telemetry data:
- Source: `src/common/metrics.rs` - Metric formatting and export
- Telemetry: `src/common/telemetry.rs` - Data collection
- Endpoint: `src/actix/api/service_api.rs:89` - HTTP handler

Metrics use `DetailsLevel::Level4` with histograms enabled for maximum detail.
