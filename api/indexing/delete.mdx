---
title: Delete Field Index
openapi: DELETE /collections/{collection_name}/index/{field_name}
---

Delete a payload index for a specific field. This removes the index structure while preserving the payload data itself.

## Use Cases

- Free up memory by removing unused indexes
- Remove indexes before changing field data types
- Clean up indexes that are no longer needed for queries
- Reduce storage overhead in on-disk configurations

## Path Parameters

<ParamField path="collection_name" type="string" required>
  Name of the collection containing the indexed field.
</ParamField>

<ParamField path="field_name" type="string" required>
  Name of the payload field for which to delete the index. For nested fields, use dot notation (e.g., `metadata.location`).
</ParamField>

## Query Parameters

<ParamField query="wait" type="boolean" default="true">
  If `true`, wait for the index deletion to complete before returning. If `false`, the operation runs in the background.
</ParamField>

<ParamField query="ordering" type="string">
  Write ordering guarantees. Options: `weak`, `medium`, `strong`.
</ParamField>

<ParamField query="timeout" type="integer">
  Operation timeout in seconds.
</ParamField>

## Examples

### Delete Simple Index

```bash
DELETE /collections/my_collection/index/city
```

### Delete Index from Nested Field

```bash
DELETE /collections/my_collection/index/metadata.location
```

### Delete Index Without Waiting

```bash
DELETE /collections/my_collection/index/description?wait=false
```

### Using cURL

```bash
curl -X DELETE \
  'http://localhost:6333/collections/my_collection/index/city?wait=true'
```


## Behavior

- The index structure is removed from memory/disk
- **Payload data is preserved** - only the index is deleted
- Points retain their payload values
- Queries on the field will still work but may be slower
- Memory/disk space used by the index is freed
- The operation cannot be undone - recreate the index if needed

## Performance Impact

### After Index Deletion

- **Queries with filters** on this field become slower
- **Memory usage** decreases
- **Disk usage** decreases (for on-disk indexes)
- **Insert/update operations** may become slightly faster (no index updates)

### Example Impact

| Scenario | With Index | After Deletion |
|----------|-----------|----------------|
| Filter query on field | < 10ms | 100-1000ms+ |
| Memory usage | Higher | Lower |
| Point insertion speed | Slightly slower | Slightly faster |

## When to Delete an Index

**Good reasons to delete:**
- Field is no longer used in filters
- Switching to a different index type
- Reducing memory consumption
- Field data is being restructured

**Keep the index if:**
- Field is actively used in search filters
- Query performance is critical
- Collection has many points (>100k)

## Error Cases

- Returns error if the field is not indexed
- Returns error if the field name doesn't exist
- No error if collection is empty

## Recreating an Index

If you delete an index and later need it again:

```json
PUT /collections/my_collection/index

{
  "field_name": "city",
  "field_schema": "keyword"
}
```

The index will be rebuilt from existing payload data.
