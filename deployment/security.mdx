Qdrant provides multiple security features to protect your vector database in production.

<Warning>
**Default Deployment is Insecure**

Qdrant starts without authentication by default. Never expose an unsecured instance to public networks.
</Warning>

## Authentication Methods

Qdrant supports three authentication methods:

1. **API Keys** - Simple static key authentication
2. **JWT RBAC** - Fine-grained role-based access control
3. **mTLS** - Mutual TLS certificate authentication

### API Key Authentication

The simplest authentication method using static API keys.

#### Configuration

**Via YAML config:**

```yaml
service:
  api_key: your-secret-api-key-here
  read_only_api_key: read-only-key-here
  alt_api_key: alternative-key-for-rotation
```

**Via environment variables:**

```bash
QDRANT__SERVICE__API_KEY=your-secret-api-key
QDRANT__SERVICE__READ_ONLY_API_KEY=read-only-key
QDRANT__SERVICE__ALT_API_KEY=alternative-key
```

**Via Docker:**

```bash
docker run -p 6333:6333 \
  -e QDRANT__SERVICE__API_KEY=your-secret-api-key \
  -e QDRANT__SERVICE__ENABLE_TLS=true \
  qdrant/qdrant
```

#### Using API Keys

Include the API key in the `api-key` header:

```bash
curl -X GET 'http://localhost:6333/collections' \
  -H 'api-key: your-secret-api-key'
```

With Python client:

```python
from qdrant_client import QdrantClient

client = QdrantClient(
    url="https://your-qdrant-instance.com",
    api_key="your-secret-api-key"
)
```

#### Read-Only API Keys

Provide limited access for read operations:

```bash
curl -X GET 'http://localhost:6333/collections/my_collection' \
  -H 'api-key: read-only-key'
```

Write operations will be rejected:

```bash
curl -X PUT 'http://localhost:6333/collections/new_collection' \
  -H 'api-key: read-only-key'  # Returns 403 Forbidden
```

#### API Key Rotation

Use `alt_api_key` for zero-downtime key rotation:

1. Set new key as `alt_api_key`
2. Update clients to use new key
3. Move `alt_api_key` to `api_key`
4. Remove old `alt_api_key`

```yaml
service:
  api_key: old-key
  alt_api_key: new-key  # Both keys work
```

After migration:

```yaml
service:
  api_key: new-key  # Only new key works
```

### JWT RBAC (Role-Based Access Control)

Advanced authentication with fine-grained permissions using JWT tokens.

#### Enable JWT RBAC

```yaml
service:
  api_key: your-jwt-secret-key  # Used to sign/verify JWTs
  jwt_rbac: true
```

<Note>
**API Key Size for JWT**

When using JWT RBAC, use an API key of at least 32 bytes (256 bits) for HMAC-SHA256.
</Note>

Environment variables:

```bash
QDRANT__SERVICE__API_KEY=your-32-byte-secret-key-here
QDRANT__SERVICE__JWT_RBAC=true
```

#### Generate JWT Tokens

Create a JWT with specific permissions:

```python
import jwt
from datetime import datetime, timedelta

secret_key = "your-32-byte-secret-key-here"

payload = {
    "exp": datetime.utcnow() + timedelta(days=30),
    "value_exists": {  # Required field
        "access": "r",  # Read-only
        "collection": "my_collection"
    }
}

token = jwt.encode(payload, secret_key, algorithm="HS256")
print(token)
```

Permission levels:
- `r` - Read-only access
- `w` - Read and write access
- `m` - Full management access (create/delete collections)

#### Use JWT Tokens

```bash
curl -X GET 'http://localhost:6333/collections/my_collection' \
  -H 'api-key: eyJhbGc...'  # JWT token
```

#### JWT Permissions Examples

**Global read access:**

```json
{
  "exp": 1735689600,
  "value_exists": {
    "access": "r"
  }
}
```

**Write access to specific collection:**

```json
{
  "exp": 1735689600,
  "value_exists": {
    "access": "w",
    "collection": "products"
  }
}
```

**Management access (create/delete collections):**

```json
{
  "exp": 1735689600,
  "value_exists": {
    "access": "m"
  }
}
```

## TLS/HTTPS Encryption

Encrypt client-to-server communication.

### Generate TLS Certificates

#### Self-Signed Certificate (Development)

```bash
mkdir -p tls
openssl req -x509 -newkey rsa:4096 -keyout tls/key.pem -out tls/cert.pem \
  -days 365 -nodes -subj '/CN=localhost'
```

#### Production Certificate (Let's Encrypt)

```bash
certbot certonly --standalone -d qdrant.example.com

# Copy certificates
cp /etc/letsencrypt/live/qdrant.example.com/fullchain.pem tls/cert.pem
cp /etc/letsencrypt/live/qdrant.example.com/privkey.pem tls/key.pem
```

### Enable TLS

```yaml
service:
  enable_tls: true

tls:
  cert: ./tls/cert.pem
  key: ./tls/key.pem
  cert_ttl: 3600  # Reload cert every hour
```

Environment variables:

```bash
QDRANT__SERVICE__ENABLE_TLS=true
QDRANT__TLS__CERT=/qdrant/tls/cert.pem
QDRANT__TLS__KEY=/qdrant/tls/key.pem
```

Docker:

```bash
docker run -p 6333:6333 \
  -e QDRANT__SERVICE__ENABLE_TLS=true \
  -e QDRANT__SERVICE__API_KEY=your-api-key \
  -v $(pwd)/tls:/qdrant/tls:ro \
  qdrant/qdrant
```

### Access via HTTPS

```bash
curl https://localhost:6333/collections \
  -H 'api-key: your-secret-api-key'
```

With Python client:

```python
client = QdrantClient(
    url="https://qdrant.example.com",
    api_key="your-secret-api-key",
    https=True
)
```

### Mutual TLS (mTLS)

Require client certificates for authentication.

#### Enable Client Certificate Verification

```yaml
service:
  enable_tls: true
  verify_https_client_certificate: true

tls:
  cert: ./tls/server-cert.pem
  key: ./tls/server-key.pem
  ca_cert: ./tls/ca.pem  # CA to verify client certs
```

#### Generate Client Certificates

```bash
# Create CA
openssl genrsa -out ca.key 4096
openssl req -new -x509 -days 365 -key ca.key -out ca.pem

# Create client cert signed by CA
openssl genrsa -out client-key.pem 4096
openssl req -new -key client-key.pem -out client.csr
openssl x509 -req -days 365 -in client.csr -CA ca.pem -CAkey ca.key \
  -set_serial 01 -out client-cert.pem
```

#### Use with Client Certificate

```bash
curl https://localhost:6333/collections \
  --cert client-cert.pem \
  --key client-key.pem \
  --cacert ca.pem
```

## Cluster Security

### TLS for P2P Communication

Encrypt internal cluster communication:

```yaml
cluster:
  enabled: true
  p2p:
    port: 6335
    enable_tls: true

tls:
  cert: ./tls/cert.pem
  key: ./tls/key.pem
  ca_cert: ./tls/ca.pem  # Required for P2P TLS
```

Environment variables:

```bash
QDRANT__CLUSTER__ENABLED=true
QDRANT__CLUSTER__P2P__ENABLE_TLS=true
QDRANT__TLS__CERT=/qdrant/tls/cert.pem
QDRANT__TLS__KEY=/qdrant/tls/key.pem
QDRANT__TLS__CA_CERT=/qdrant/tls/ca.pem
```

### Network Segmentation

Isolate cluster traffic:

```yaml
# Public-facing nodes
service:
  host: 0.0.0.0  # Accept external connections
  http_port: 6333

cluster:
  p2p:
    port: 6335  # Internal network only
```

Firewall rules:

```bash
# Allow HTTP API from public
iptables -A INPUT -p tcp --dport 6333 -j ACCEPT

# Allow P2P only from cluster nodes
iptables -A INPUT -p tcp --dport 6335 -s 10.0.0.0/24 -j ACCEPT
iptables -A INPUT -p tcp --dport 6335 -j DROP
```

## Docker Security

### Run as Non-Root User

```bash
docker run -p 6333:6333 \
  --user=1000:2000 \
  -e QDRANT__SERVICE__API_KEY=your-api-key \
  qdrant/qdrant
```

Docker Compose:

```yaml
services:
  qdrant:
    image: qdrant/qdrant:latest
    user: "1000:2000"
    environment:
      - QDRANT__SERVICE__API_KEY=your-api-key
```

### Read-Only Filesystem

```bash
docker run -p 6333:6333 \
  --read-only \
  --tmpfs /tmp \
  -v $(pwd)/storage:/qdrant/storage \
  -v $(pwd)/snapshots:/qdrant/snapshots \
  qdrant/qdrant
```

### Limit Resources

```yaml
services:
  qdrant:
    image: qdrant/qdrant:latest
    deploy:
      resources:
        limits:
          cpus: '4'
          memory: 8G
          pids: 1000
        reservations:
          cpus: '2'
          memory: 4G
```

### Drop Capabilities

```bash
docker run -p 6333:6333 \
  --cap-drop=ALL \
  --cap-add=NET_BIND_SERVICE \
  qdrant/qdrant
```

## Kubernetes Security

### Security Context

```yaml
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: qdrant
spec:
  template:
    spec:
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000
        fsGroup: 2000
        seccompProfile:
          type: RuntimeDefault
      containers:
      - name: qdrant
        image: qdrant/qdrant:latest
        securityContext:
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
          capabilities:
            drop:
            - ALL
```

### Network Policies

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: qdrant-network-policy
spec:
  podSelector:
    matchLabels:
      app: qdrant
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: application
    ports:
    - protocol: TCP
      port: 6333
  - from:
    - podSelector:
        matchLabels:
          app: qdrant
    ports:
    - protocol: TCP
      port: 6335  # P2P only from other Qdrant pods
```

### Pod Security Standards

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: qdrant
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/warn: restricted
```

## Audit Logging

Track all API access for security compliance:

```yaml
audit:
  enabled: true
  dir: /var/log/qdrant/audit
  rotation: daily
  max_log_files: 30
  trust_forwarded_headers: false  # Only enable with trusted proxy
```

Environment variables:

```bash
QDRANT__AUDIT__ENABLED=true
QDRANT__AUDIT__DIR=/var/log/qdrant/audit
```

Audit log format (JSON):

```json
{
  "timestamp": "2026-03-03T10:15:30Z",
  "client_ip": "192.168.1.100",
  "method": "PUT",
  "path": "/collections/my_collection/points",
  "user_agent": "qdrant-client/1.0",
  "status": 200,
  "duration_ms": 45
}
```

## Reverse Proxy Security

### NGINX with TLS Termination

```nginx
server {
    listen 443 ssl http2;
    server_name qdrant.example.com;
    
    ssl_certificate /etc/letsencrypt/live/qdrant.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/qdrant.example.com/privkey.pem;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    
    # Rate limiting
    limit_req_zone $binary_remote_addr zone=qdrant:10m rate=10r/s;
    limit_req zone=qdrant burst=20 nodelay;
    
    location / {
        proxy_pass http://localhost:6333;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        
        # Forward API key
        proxy_set_header api-key $http_api_key;
    }
}
```

## Security Checklist

Before production deployment:

- [ ] Enable API key authentication
- [ ] Use TLS/HTTPS for all connections
- [ ] Generate strong API keys (32+ bytes for JWT)
- [ ] Enable TLS for cluster P2P communication
- [ ] Run containers as non-root user
- [ ] Use read-only filesystem where possible
- [ ] Set resource limits (CPU, memory)
- [ ] Enable audit logging
- [ ] Configure network policies/firewall rules
- [ ] Restrict port exposure (bind to localhost if behind proxy)
- [ ] Set up monitoring and alerting
- [ ] Regular security updates
- [ ] Backup encryption keys and certificates
- [ ] Document incident response procedures
- [ ] Regular security audits

## Best Practices

1. **Never expose Qdrant without authentication** to public networks
2. **Always use TLS/HTTPS** in production
3. **Rotate API keys regularly** using `alt_api_key`
4. **Use strong random keys** (at least 32 bytes)
5. **Enable audit logging** for compliance
6. **Separate network segments** for cluster traffic
7. **Use JWT RBAC** for fine-grained access control
8. **Monitor authentication failures** for suspicious activity
9. **Keep Qdrant updated** with latest security patches
10. **Use secrets management** (Kubernetes Secrets, Vault, etc.)
